TIMING LOGIC & RECORDING
========================

1. CLOCK CALIBRATION (once, at session start)
----------------------------------------------
When both players have joined, Firebase writes a single server-side
experimentStartTime timestamp. Each player reads it and computes:

    clientServerTimeDiff = Firebase server time - player's local clock time

This tells each player how far their local clock is from the server clock.
It is saved in the CSV footer as:  # client_server_diff_ms

Both players share the same Firebase timestamp as their reference point,
so their key-press times can be compared even if their machines disagree.


2. IMAGE LOAD GATE (once, per player)
--------------------------------------
After both players have received the trial sequence, each player preloads
all chart and symbol images and writes player{N}_images_loaded = true to
Firebase. Once BOTH flags are set, each player:

  a. Resets trialSyncTime to null and their own player{N}_ready to false
     (clears stale data from any previous run that could cause a false-start)
  b. Sets experimentWallStartTime = Date.now()  (local "zero point" for CSV)
  c. Attaches the keyboard handler and starts the game loop at 'instructions'

This ensures both players enter the instructions phase at the same time
and with a clean Firebase state.


3. SPACE-PRESS SYNC (once, when players press SPACE)
-----------------------------------------------------
The instructions phase waits for both players to press SPACE before the
first trial begins. The sync protocol:

  a. Each player presses SPACE and writes player{N}_ready = true to Firebase.
  b. Once both flags are set, Player 1 writes:
         trialSyncTime = firebase.firestore.FieldValue.serverTimestamp()
  c. Both players listen for trialSyncTime via an onSnapshot listener.
  d. When the snapshot fires (with the server-confirmed value), each player
     computes:
         syncedPhaseStartTime = Date.now() + 1000ms
     and schedules a setTimeout(1000ms) that sets bothPlayersPressedSpace = true.
  e. On the next game-loop frame after bothPlayersPressedSpace is true,
     startPhase('baseline') fires. Because startPhase uses syncedPhaseStartTime
     as the anchor on the very first call (instead of Date.now()), both devices
     share an identical phaseStartTime derived from the same moment in time.

The residual skew between the two devices is just the difference in Firestore
notification latency (~10–50 ms), far smaller than the previous approach which
accumulated up to ~1000 ms of error from independent Date.now() calls.

Guard: the trialSyncReceived flag ensures that if Firestore fires the snapshot
twice (provisional local write, then server-confirmed value), only the first
fire sets syncedPhaseStartTime.


4. FIXED-INCREMENT PHASE TIMING (every phase transition)
---------------------------------------------------------
Within a trial, phases advance using fixed-increment timing rather than
Date.now() snapshots:

    phaseStartTime = phaseStartTime + configuredDurationOfPreviousPhase

The configured durations (in ms):
    baseline:          1000
    sample:            1000
    delay:             1000
    decision:          2000
    feedback:          2000
    postFeedbackDelay: 1000

This prevents per-frame overshoots (~16 ms at 60 fps) from accumulating
into drift across hundreds of trials. Player-triggered phases ('instructions',
'lottery') have no fixed duration, so they reset phaseStartTime to Date.now()
when they start.


5. PER-TRIAL DECISION TIMING (every trial)
------------------------------------------
When the player presses a key during the decision phase:

    decisionTimestampMs = Date.now()  (exact moment of key press, local clock)

Three timing values are computed and saved per CSV row:

    elapsed_ms         = decisionTimestampMs - experimentWallStartTime
                         Time from experiment start to key press, in this
                         player's local clock. NOT directly comparable
                         across players.

    server_elapsed_ms  = elapsed_ms + clientServerTimeDiff
                         Same moment, but expressed in server time.
                         Both players' values are in the same coordinate
                         system, so you CAN directly compare them across
                         the two CSVs to see who decided first and by
                         how much.

    reaction_time_ms   = decisionTimestampMs - decisionPhaseStartTime
                         Time from when the decision phase began to when
                         the key was pressed (response time).


6. SERVER_TIMESTAMP (async, per trial)
---------------------------------------
When a decision is made, it is uploaded to Firebase. Firebase assigns a
server-side timestamp when it receives the upload. Once confirmed, the code
patches that timestamp into the log entry:

    server_timestamp = Firebase receipt time (ISO string)

This is NOT the exact moment of key press — it includes network latency.
Its main use is as an absolute audit trail and for the competition condition
(symbol 3), where both players' upload timestamps are compared to resolve
ties (first-to-upload wins).


7. CSV COLUMNS SUMMARY
-----------------------
Per-trial columns (in order):

    trial               Trial number
    phase               Experiment phase (1, 2, 3; 0 = catch trial)
    elapsed_ms          Key press time from experiment start (local clock)
    server_elapsed_ms   Key press time from experiment start (server clock)
                        — use this to compare decisions across players
    reaction_time_ms    Key press time from decision phase start (RT)
    server_timestamp    Absolute Firebase timestamp of upload (ISO string)
    condition           anticoordination / coordination / competition / catch
    delta               Point difference ratio for the trial's chart
    scale               Total points available (S value)
    chartId             Which chart configuration was shown
    point1_color        Color of point 1 shown during sample (always red)
    point1_location     Screen location of point 1 (up/down/left/right)
    point1_value        Point value of point 1 (the larger amount)
    point2_color        Color of point 2 shown during sample (always blue)
    point2_location     Screen location of point 2
    point2_value        Point value of point 2 (the smaller amount)
    choice              Direction key the player pressed (or 'none')
    your_points         Points this player earned on the trial
    partner_points      Points the other player earned on the trial

Footer metadata:

    # experiment_start          Local wall clock time experiment began
    # client_server_diff_ms     This player's local clock offset from server
    # total_points              Sum of your_points across all trials
    # total_trials              Number of trials logged


8. HOW TO VERIFY SYNC BETWEEN PLAYERS
---------------------------------------
Load both players' CSVs. For the same trial number, compare server_elapsed_ms.
The difference tells you how many milliseconds apart their key presses were,
in a common time reference. A small difference (< 2000ms) is expected since
both players have the same decision window.

To check how well the clocks were synced, compare each player's
client_server_diff_ms — a large difference between the two players' offsets
would indicate one player had a poor connection at session start.

If sync breaks, check:
  - trialSyncTime is reset to null in waitForBothPlayersImagesLoaded
    (prevents stale data causing an immediate false-start on re-run)
  - clientServerTimeDiff is computed correctly in startExperiment
  - syncedPhaseStartTime is set in waitForTrialSyncAndStart and consumed
    (and then nulled) in startPhase('baseline')
  - trialSyncReceived guard is reset to false when entering 'instructions'
